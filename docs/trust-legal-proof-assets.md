# Konative Trust, Legal, and Proof Assets

## Purpose
This document turns the abstract trust layer from `docs/DESIGN-DOC.md` into a concrete v1 asset list for the ASAP public launch.

The goal is not to create a giant brand library before shipping. The goal is to identify the minimum asset set required to make `konative.com` feel credible, professional, and safe to use for active deal flow.

## Launch Posture
The v1 trust strategy is:

- **Konative-only public brand**
- **Competence-first trust**
- **Credentials plus anonymized examples**
- **No dependency on named public case studies**
- **ASAP release bias**

## Trust Layer Strategy
For the first public release, trust should come from four things:

1. **Clear problem understanding**
2. **Specific buyer and market language**
3. **Visible operator credibility**
4. **Professional legal/contact posture**

The site does not need to look giant. It needs to look serious, coherent, and real.

## Minimum Viable Trust Assets For V1

## 1. Founder / leadership credibility
### Required assets
- Founder name
- Short founder bio
- Relevant industry/operator background
- Optional headshot if available and professional enough

### Website use
- trust section
- about/founder block
- contact or CTA destination

### Notes
If there is a broader bench behind Konative, the site can mention specialist coordination without pretending a full visible team page exists yet.

## 2. Market credibility cues
### Required assets
- Clear statement of buyer focus:
  - investment groups
  - Indigenous Development Corporations
- Clear statement of market urgency:
  - capital is available
  - land is available
  - power/supply-chain delays create timing risk

### Website use
- hero
- urgency section
- who-we-serve section

### Notes
These are copy assets, but they function as trust assets because specificity signals legitimacy.

## 3. Anonymized proof assets
### Recommended v1 proof formats
Use one or more of the following:

1. **Sample deliverable excerpt**
   - sanitized risk-register page
   - sanitized executive-readout page
   - sanitized project-readiness framework

2. **Example engagement structure**
   - kickoff
   - diligence workstreams
   - decision package
   - executive readout

3. **Anonymized project pattern**
   - buyer type
   - geography type
   - urgency driver
   - problem pattern
   - what the readiness engagement clarified

### Website use
- trust section
- engagement section
- optional downloadable example later

### Recommended first proof asset
**Start with a sanitized sample deliverable structure**, not a pseudo-case-study.

Best candidates:
- one-page sample risk register structure
- one-page sample executive readout structure
- one-page sample project-readiness framework

This is safer and faster than trying to invent an anonymized case study that sounds more specific than the permissions allow.

## 4. Geography and domain specificity
### Required assets
- brief reference to Canada / rural / Indigenous development context where relevant
- clear mention of modular data center development readiness
- terminology that proves real familiarity with:
  - site viability
  - power path
  - cooling/infrastructure
  - supply-chain critical path

### Website use
- hero support line
- what-we-do section
- trust section

## 5. Contact credibility
### Required assets
- professional email address on the Konative domain
- clear CTA destination
- visible contact pathway if the form fails

### Website use
- CTA destination page
- footer
- final CTA band

## Legal and Privacy Assets For V1

## Required pages
1. **Privacy Policy**
2. **Contact page / CTA page**
3. **Legal / Terms page** if needed for the site and engagement inquiry flow

### Privacy policy should cover
- what the site collects through the inquiry form
- why the information is collected
- how follow-up communication happens
- any analytics or tracking used on the site

### Contact/legal posture should cover
- Konative public branding
- contact channel
- any necessary operating/legal disclosure for the business

### Notes
Because the public brand is `Konative only`, legal text should stay clean and confidence-building. Avoid cluttering the homepage with legal structure details unless required; keep the detail in footer-linked pages.

## Form Trust Assets
These are small, but they matter:

- form heading that reinforces seriousness
- short explanation of who the review is for
- minimum-field form that feels professional, not invasive
- visible success state
- fallback contact method if form submission fails

## Recommended Copy Signals In The Trust Layer
Use signals like:

- decision-grade diligence
- development readiness
- go / no-go clarity
- qualified projects
- project readiness review

Avoid signals like:

- revolutionary platform
- end-to-end consulting solutions
- generic innovation language
