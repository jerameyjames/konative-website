# Konative.com — Founder OS operating model

This document applies **Founder OS** thinking (vision → goals → projects → execution, with meetings feeding tasks and metrics) to the Konative web property and product work. Use it as the spine for your Notion workspace: duplicate section headings as top-level or child pages, or as rows in your Projects / Tasks databases.

## 1. North star

| Field | Content (fill in / refine) |
|--------|----------------------------|
| One-line vision | *What Konative exists to do for whom* |
| Primary outcome | *Measurable outcome for customers (e.g. time saved, revenue, risk reduced)* |
| Non-goals | *What we will not build this quarter* |

## 2. Strategic pillars (3–5)

Short labels that every initiative should roll up to—for example: **Trust**, **Clarity**, **Velocity**, **Sustainable GTM** (adjust to your actual strategy).

## 3. Goals & horizons

- **Now (0–30d):** launch-quality bar for konative.com, core messaging, primary CTA path.
- **Next (30–90d):** experiments, content/campaigns, integrations or features tied to north star.
- **Later:** captured as backlog ideas, not active commitments.

## 4. Workstreams (Founder OS lanes)

| Lane | Purpose | Owner | Cadence |
|------|---------|-------|---------|
| Product & web | Site, product UX, technical delivery | TBD | Weekly |
| GTM & content | Messaging, SEO, campaigns, partnerships | TBD | Weekly |
| Operations | Tools, process, vendor, legal basics | TBD | Monthly |
| Finance & metrics | Unit economics, reporting, dashboards | TBD | Monthly |

## 5. Operating rhythm

- **Weekly:** priorities (3–5), blockers, ship list for konative.com + product.
- **Monthly:** pillar review, KPIs, retire or promote backlog items.
- **Meetings → execution:** decisions and tasks captured same day (see `requirements-prd.md` for backlog shape).

## 6. Notion object model (recommended)

Mirror common Founder OS / spec-to-implementation patterns:

1. **Hub page:** `Konative — Founder OS` (links to everything below).
2. **PRD / Requirements** (this repo: `docs/requirements-prd.md`) — single source for scope; Notion page can summarize and link to GitHub or this folder.
3. **Projects database:** Name, Status, Priority, Target date, Link to PRD section, Owner.
4. **Tasks database:** Title, Status, Priority, Project relation, Due, Acceptance notes.
5. **Decisions log** (optional): Date, Decision, Context, Links to tasks.

## 7. AI / agent workflow (skills alignment)

- **Specs → build:** Use Notion + repo PRD together: agents read `docs/requirements-prd.md` for acceptance criteria; Notion tracks status and owners.
- **Research / briefs:** Pull multiple sources into one brief before changing scope.
- **Shipping:** Keep changelog and version discipline when code lives in this repo.
