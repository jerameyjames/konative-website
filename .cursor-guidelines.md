## Stack Context

- **Language/Framework:** Node.js, React, Next.js 14
- **Database:** Cloud SQL (PostgreSQL)
- **Infra:** Vercel (hosting), Prisma ORM
- **CI/CD:** GitHub Actions → staging (auto-deploy) → production (manual trigger)
- **Key constraint:** Staging auto-deploys; production requires manual verification before deploy. Database schema changes are high-risk.

---

## Tier 1: SHIP IT (No approval needed)

Own these completely. Self-review, merge, done.

✅ Bug fixes in existing features  
✅ Refactoring / code cleanup (no logic change)  
✅ Patch/minor dependency updates  
✅ UI/UX improvements (no API change)  
✅ Copy, text, error messages, styling changes  
✅ Test additions (unit, integration)  
✅ Documentation updates  
✅ CI/CD workflow improvements (non-breaking)  
✅ Monitoring / observability additions  
✅ Client-side configuration changes  

---

## Tier 2: PR + HEADS-UP (Async review, not blocking)

Open a PR with a clear description. If no response in **3 business days**, merge with confidence.

❓ New features (significant ones)  
❓ Refactors that change how components interact  
❓ API route changes or new endpoints  
❓ Changes to authentication/authorization logic  
❓ New dependencies (include justification)  
❓ Prisma schema changes (non-breaking migrations)  
❓ Vercel deployment config changes  
❓ Environment variable additions  

**PR description template:**
```markdown
## What
[What did you build or change?]

## Why
[Problem solved or opportunity]

## Implementation
[Key decisions made?]

## Testing
[What did you verify locally?]

## Deployment notes
[Any special steps needed for staging/production?]
```

---

## Tier 3: CREATE ISSUE (Wait for explicit decision)

Do NOT start coding. Create a GitHub issue and wait for go/no-go.

🚫 Major architecture changes (API redesign, data model overhaul)  
🚫 Breaking database schema changes  
🚫 Changes to authentication/security model  
🚫 Significant performance optimizations affecting infrastructure  
🚫 Third-party service integrations  

---

## Decision Tree
```
Is it a bug fix or style/refactor?
└─ YES → Tier 1. Ship it.

Does it change how the system works (API, schema, auth)?
└─ YES → Tier 2. PR + heads-up.

Does it change architecture or security?
└─ YES → Tier 3. Issue first, wait for decision.

Unsure?
└─ Default to Tier 2. Open PR.
```

## Project-Specific Rules

- Always test database migrations locally before opening PR (`npm run db:migrate`)
- Staging auto-deploys on merge to `main`; wait for staging to be green before deploying to production
- Production deployments require manual trigger via GitHub Actions—never merge and auto-deploy to prod
- Schema changes must include a migration file; document breaking changes clearly
- Environment variables for staging and production must be kept in sync

---

## Production Emergency (Break Glass)

Site is down or critically broken:
```
1. git checkout -b hotfix/describe-the-issue
2. Fix + test thoroughly locally
3. git commit -m "hotfix: describe the fix"
4. git push origin hotfix/describe-the-issue
5. Open PR, merge immediately
6. Wait for staging deploy to succeed
7. Manually trigger production deploy
8. Notify stakeholders
9. Create postmortem issue after stability returns
```

---

## Never (Hard Rules)

🚫 Commit secrets, API keys, or passwords to git  
🚫 Deploy to production without testing in staging first  
🚫 Skip local database migration testing before opening PR  
🚫 Introduce new major version dependencies without review  
🚫 Modify production data without explicit Tier 3 approval  
🚫 Push directly to main—always use feature branches and PRs
